Docker 架構：Jenkins Container 如何透過 docker.sock 操作 Host Docker

學習日期：2026-07-01
主題：Docker CLI、Docker daemon、Unix socket，以及 Jenkins Container 與 Host Docker 的關係

學習目標

  • 理解 Docker CLI 與 Docker daemon 的分工。
  • 理解 Host Docker、docker.sock 與 Unix socket。
  • 理解 Jenkins Container 如何操作 Host Docker。
  • 理解 bind mount 為什麼要從 Docker Host 的視角看路徑。

Docker 的基本架構

執行 docker run nginx 時，真正的流程是：

Docker 元件 餐廳類比
Docker CLI 點餐櫃台
Docker API 訂單格式
Docker daemon 廚房
Image 食譜
Container 按照食譜製作的餐點
Host 餐廳所在的建築物
docker.sock 櫃台傳送訂單到廚房的窗口

Host Docker 是什麼？

Host 是實際執行 Container 的機器，可能是實體 Linux Server、Linux VM、EC2，或 Docker Desktop 背後的 Linux VM。

Host Docker 通常是指運行在 Host 上的 Docker daemon，以及由它管理的 Docker 環境。

這種模式稱為 Docker-outside-of-Docker，不是 Docker-in-Docker。

Docker daemon 是什麼？

Daemon 是長時間在背景運行並等待請求的程式。Docker daemon 通常叫做 dockerd，負責：

  • 下載與管理 Image
  • 建立、啟動及停止 Container
  • 管理 Network 與 Volume
  • 處理 bind mount
  • 接收 Docker CLI 傳來的 API 請求

docker ps 並不是 Docker CLI 自己尋找 Container，而是 CLI 詢問 daemon，再顯示 daemon 的回應。

核心觀念：

Docker CLI 負責下指令，Docker daemon 才是真正執行工作的人。

Unix socket 是什麼？

Unix socket 是同一台 Unix/Linux 機器上，兩個 Process 互相通訊的一種方式。

通訊方式 位址範例
TCP socket 127.0.0.1:2375
Unix socket /var/run/docker.sock

Jenkins 執行 Docker 指令的完整流程

  1. Jenkins Container 裡的 Docker CLI 讀取 Compose 設定。
  2. CLI 透過 docker.sock 將請求送給 Host Docker daemon。
  3. daemon 在 Host Docker 環境建立 worker Container。
  4. daemon 設定 Network、Volume 與 bind mount。
  5. daemon 啟動 Container 裡的 Process。

因此：

指令從 Jenkins Container 發出，但真正建立 Container 的是 Host Docker daemon。

安全性提醒

能存取 /var/run/docker.sock，通常就接近擁有整台 Docker Host 的高權限，因為它可以：

  • 啟動或刪除其他 Container
  • 掛載 Host 目錄
  • 建立高權限 Container
  • 存取其他 Container 的部分資源

因此 Jenkins Pipeline 與使用者權限必須受到嚴格管理。

重點整理

  1. Docker CLI 只負責傳送指令。
  2. Docker daemon 負責建立及管理 Container。
  3. docker.sock 是 CLI 與本機 daemon 的通訊入口。
  4. Unix socket 使用檔案路徑作為同機 Process 的通訊端點。
  5. Jenkins 透過掛載 Host 的 docker.sock 操作 Host Docker。
  6. Jenkins 與它啟動的工作 Container 是兄弟關係。
  7. Bind mount 來源路徑必須從 Docker daemon 所在 Host 的視角理解。
  8. docker cp 可以避免依賴 Jenkins Workspace 在 Host 上的實體路徑。

自我檢查

Docker CLI 會直接建立 Container 嗎？

不會。CLI 會透過 Docker API 將請求送給 daemon，由 daemon 建立 Container。

Jenkins Container 為什麼能執行 docker ps？

因為 Host 的 docker.sock 被掛載進 Jenkins Container，讓裡面的 CLI 可以與 Host daemon 溝通。

Jenkins 建立的工作 Container 是 Jenkins Container 的子 Container 嗎？

不是。它們都是 Host Docker daemon 管理的兄弟 Container。

Bind mount 左側的路徑由誰解析？

由 Docker daemon 所在的 Host 解析。

為什麼改用 docker cp？

它能從 Container 取出檔案，再由 Jenkins Container 裡的 CLI 寫入 $WORKSPACE，不需要知道 Workspace 在 Host 上的實際路徑。
